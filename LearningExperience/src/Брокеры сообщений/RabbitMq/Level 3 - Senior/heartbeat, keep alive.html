Сеть может выйти из строя по-разному, иногда довольно незаметно (например, при большой потере пакетов). Нарушенные TCP-соединения могут быть обнаружены операционной системой через умеренно долгое время (например, около 11 минут с конфигурацией по умолчанию в Linux). AMQP 0-9-1 предлагает функцию тактового сигнала (heartbeat), чтобы гарантировать, что уровень приложения быстро узнает о нарушенных соединениях (а также о полностью не отвечающих одноранговых узлах). Heartbeats также защищают от определенного сетевого оборудования, которое может прервать "простаивающие" TCP-соединения, если на нем нет активности в течение определенного периода времени.

Протоколы активности TCP (keepalives)> - это функция стека TCP, которая служит той же цели и может быть очень полезной (возможно, в сочетании с тактовыми сигналами), но требует настройки ядра, чтобы быть практичной с большинством операционных систем и дистрибутивов.

Heartbeat

Значение heartbeat определяет, по истечении какого периода времени каждое TCP-соединение должно считаться недоступным (отключенным) для RabbitMQ и клиентских библиотек. Это значение согласовывается между клиентом и сервером RabbitMQ во время соединения. Клиент должен быть настроен на запрос heartbeats.
Процесс согласования работает следующим образом: сервер предложит свое настраиваемое значение, а клиент свяжет его со своим настроенным значением и отправит полученное значение обратно. Значение указано в секундах, а значение по умолчанию, предложенное RabbitMQ, равно 60.

Keep alive

TCP содержит механизм, схожий по назначению с heartbeats (keepalive) в протоколах обмена сообщениями и тайм-аутом сетевого тика, описанном выше: TCP keepalive. Из-за неадекватных значений по умолчанию нельзя предполагать, что пакеты TCP keepalives подходят для протоколов обмена сообщениями. Однако при правильной настройке они могут быть полезны в качестве дополнительного механизма защиты в средах, где нельзя ожидать, что приложения будут разрешать heartbeats или использовать разумные значения.
